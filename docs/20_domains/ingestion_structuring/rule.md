# Rule-Based 4요소 구조화 설계안 (강화 버전)

## 1. 목적

본 문서는 민원 텍스트를 4요소(관찰, 요청, 맥락, 결과)로 구조화하기 위한 고신뢰 rule-based 설계를 정의한다.
목표는 다음과 같다.

- LLM 없이도 일관된 구조화 품질 확보
- Q/A 혼합 텍스트, 단문 입력, 비정형 입력 처리 안정성 개선
- 사용자 입력(답변 미존재)과 민원 DB(답변 존재)를 명확히 구분
- 근거 span, 신뢰도, 상태값을 통해 후속 검색/평가 파이프라인과 안전하게 연동

## 2. 기존 취약점 요약

현재 규칙의 주요 한계:

- Q/A 분리 취약: `Q:` / `A:` 패턴에 과의존
- 문장 분리 단순: 마침표 중심 분할로 한국어 조사/어미 문맥 손실
- result 강제 채움: 답변이 없어도 기본 문구로 채워 정보 왜곡 가능
- context 추출 약함: 제목 1줄 또는 시간/장소 키워드 1문장에 치우침
- evidence_span 부정확: 부분 문자열 중복 시 첫 매치 오탐 가능

## 3. 설계 원칙

- Deterministic first: 동일 입력은 항상 동일 출력
- Evidence first: 추출 텍스트는 반드시 원문 span 근거를 가진다
- Missing is explicit: 없는 정보는 억지 생성하지 않고 상태로 표현
- Domain-safe defaults: 민원 도메인 표현(요청/처리/회신/민원인 등)에 특화
- Progressive fallback: 강한 규칙 실패 시 약한 규칙으로 단계적 하향

## 4. 파이프라인 개요

1. 전처리/정규화
2. 세그먼트 분리 (제목/질의/답변/본문)
3. 문장 후보 생성
4. 4요소별 후보 탐지
5. 후보 스코어링 및 최종 선택
6. evidence span 정렬/검증
7. 상태값 및 confidence 산정
8. 스키마 검증

## 5. 입력 타입 분류

입력 텍스트를 먼저 3가지로 분류한다.

- TYPE_A: `Q/A` 동시 존재 (민원 DB형)
- TYPE_B: 질의/요청만 존재 (사용자 입력형)
- TYPE_C: 자유 서술형 (게시글/메모형)

분류 기준 예시:

- A 마커: `A:`, `답변`, `회신`, `처리결과`, `조치결과`
- Q 마커: `Q:`, `민원내용`, `문의`, `요청사항`
- 제목 마커: `제목:`, `[제목]`

## 6. 전처리 규칙

### 6.1 텍스트 정규화

- 유니코드 정규화(NFC)
- 개행 통일(`\r\n -> \n`)
- 연속 공백 2개 이상 축소(단, 전화번호/코드 블록 제외)
- 불필요 제어문자 제거

### 6.2 메타 잡음 제거

다음 패턴은 별도 메타로 분리하고 본문에서 제거 가능:

- 인사/종결 관용구: `감사드립니다`, `끝.`
- 연락처 블라인드 토큰: `▲▲▲`, `☎` 등
- 번호 매김 접두: `1.`, `가.`, `나.`

## 7. 세그먼트 분리 규칙

우선순위 높은 규칙부터 적용:

1. 강패턴 분리
- `Q\s*:` 이후를 question
- `A\s*:` 이후를 answer

2. 준강패턴 분리
- `민원내용`, `문의내용`, `요청사항` 블록을 question
- `답변`, `처리결과`, `조치결과`, `검토의견` 블록을 answer

3. 약패턴 분리
- 처리기관 관용 문구 시작 시 answer 후보로 간주
  - 예: `귀하께서`, `아래와 같이 답변`, `처리 결과`

분리 실패 시:
- 전체를 question으로 간주
- answer는 비어있는 상태로 유지

## 8. 문장 후보 생성 규칙

- 한국어 문장 종결 어미 및 구두점 기반 분리
  - `다.`, `요.`, `니다.`, `?`, `!`, 개행
- 불필요한 초단문(길이 < 5) 제거
- 문장 원문 인덱스(start/end) 보존

## 9. 4요소별 탐지 규칙

## 9.1 Observation (관찰)

정의: 현재 문제/현상/불편/위험에 대한 사실 서술

강신호 키워드:
- `불법`, `위반`, `무단`, `점거`, `무단점용`, `불법주차`, `차량통행`, `과속`, `정체`
- `고장`, `파손`, `균열`, `침하`, `누수`, `역류`, `범람`, `악취`, `소음`, `진동`, `분진`
- `위험`, `사고`, `화재`, `미끄럼`, `추락`, `붕괴`, `감전`, `안전사고`, `2차사고`
- `불편`, `민원`, `민원 다수`, `이용 불가`, `통행 불편`, `시야 방해`, `환경 저해`

패턴 예시:
- `~하고 있다`, `~발생`, `~지속`, `~심각`, `~문제`

제외 규칙:
- 요청 어미(`해주세요`, `요청드립니다`) 중심 문장 제외
- 기관 조치/답변 문장 제외

## 9.2 Request (요청)

정의: 민원인이 원하는 조치/행동/개선 요구

강신호 키워드:
- `요청`, `요구`, `건의`, `민원 제기`, `재요청`, `재민원`
- `조치`, `시정`, `개선`, `보수`, `정비`, `복구`, `철거`, `이전`, `교체`, `설치`, `보강`
- `점검`, `안전점검`, `현장확인`, `조사`, `원인파악`, `재발방지`, `대책 마련`
- `단속`, `계도`, `행정지도`, `행정처분`, `과태료 부과`, `원상복구 명령`, `행정대집행`
- `바랍니다`, `해주시기`, `부탁드립니다`, `검토 바랍니다`, `신속 처리`, `조속한 처리`

패턴 예시:
- `~해주시기 바랍니다`
- `~조치 요청드립니다`
- `~개선해 주세요`

선택 정책:
- 복수 후보 시 가장 구체적 행동 동사가 있는 문장 우선
- 동일 점수면 마지막 요청 문장 채택

## 9.3 Context (맥락)

정의: 시점/장소/이력/주체 등 사건 배경

강신호 키워드:
- 시간: `년`, `월`, `일`, `시`, `분`, `초`, `주`, `주말`, `평일`, `야간`, `새벽`, `출퇴근 시간`, `최근`, `지난`, `매일`, `상시`, `수시`, `지속적`
- 장소: 행정구역/도로/하천/시설명, `교차로`, `인도`, `보도`, `차도`, `골목`, `공원`, `산책로`, `하천 둔치`, `버스정류장`, `학교 앞`, `아파트 단지`
- 이력: `~회 신고`, `반복 신고`, `기존`, `이전`, `재차`, `계속`, `장기간`, `수년간`, `민원 이력`, `처리 이력`, `안전신문고`
- 주체: `민원인`, `주민`, `시민`, `보행자`, `운영단체`, `관리주체`, `담당부서`, `시청`, `구청`, `도청`

구성 전략:
- 단일 문장보다 1~2개 문장 병합 허용
- 시간+장소가 분리된 경우 병합 스코어 가산

## 9.4 Result (결과)

정의: 기관 답변/처리 상태/조치 계획/완료 내역

강신호 키워드:
- `답변드립니다`, `회신`, `안내드립니다`, `처리 결과`, `검토 결과`, `조치 결과`
- `처리`, `조치`, `시행`, `이행`, `개선`, `정비`, `보수`, `복구`, `철거`, `원상복구`
- `검토`, `협의`, `통보`, `명령`, `지시`, `계고`, `행정처분`, `과태료`, `고발`
- `완료`, `종결`, `해소`, `반영`, `접수`, `배정`, `진행`, `추진 중`, `착수`, `예정`, `계획`
- `단속 강화`, `계도 실시`, `현장 점검`, `추가 점검`, `재점검`, `후속 조치`, `협조 요청`

상태 정책(핵심):
- answer 세그먼트 존재 + 결과 패턴 검출: `status=present`
- answer 세그먼트 없음: `status=pending`
- answer 세그먼트 있으나 실질 결과 없음(인사말만): `status=insufficient`

금지 규칙:
- 결과 미존재 상황에서 임의 문구(예: "답변 본문 미제공")를 결과 텍스트로 채우지 않는다.
- 대신 빈 텍스트 + 상태값으로 표현한다.

## 10. 후보 스코어링

문장 후보 점수 공식(요소별 공통):

`score = w_kw * keyword_hits + w_pat * pattern_hits + w_seg * segment_bonus + w_len * length_score + w_pos * position_score - penalty`

권장 가중치:
- `w_kw=0.35`, `w_pat=0.25`, `w_seg=0.20`, `w_len=0.10`, `w_pos=0.10`

패널티 예시:
- 중복/상충 문장: `-0.15`
- 요소 간 동일 문장 재사용(Observation=Request 동일): `-0.10`

최종 선택:
- 요소별 최고 점수 1개(필요 시 context만 2개 병합)
- 최소 임계치 미달 시 해당 요소는 빈 텍스트 + low confidence

## 11. evidence span 정합성

- 후보 생성 시 원문 인덱스를 이미 보존하여 재탐색 오탐 방지
- 문자열 `find` 재탐색 대신 문장 토큰 기반 start/end 사용
- span 검증:
  - `0 <= start < end <= len(raw_text)`
  - `raw_text[start:end]`가 추출 text와 normalize 기준으로 동일해야 함

## 12. confidence/quality 정책

요소별 confidence:
- High: 강패턴 + 세그먼트 일치 + 키워드 2개 이상
- Mid: 키워드/패턴 일부 충족
- Low: fallback으로 채택

케이스 confidence_score:
- 4요소 평균이 아닌 가중 평균 권장
  - observation 0.30
  - request 0.30
  - context 0.25
  - result 0.15
- 단, `result.status=pending`인 TYPE_B 입력은 패널티 적용 금지

## 13. 출력 스키마 권장 변경

현재 필드를 유지하면서 result 상태를 명시하기 위해 확장:

```json
{
  "result": {
    "text": "",
    "confidence": 0.0,
    "evidence_span": [0, 0],
    "status": "pending"
  }
}
```

`status` enum:
- `present`: 결과 존재
- `pending`: 미처리/미답변
- `insufficient`: 답변은 있으나 실질 결과 부족

## 14. 예외/엣지 케이스 규칙

- 민원인이 과거 답변을 인용한 경우:
  - `"~라고 답변받음"`은 context로 우선 분류
- 질문 내 희망결과 표현(`~되었으면`)은 request로 유지
- 공지문/안내문 단독 텍스트는 result 우선, observation/request 낮춤
- 전화상담 요약(턴 데이터)은 턴별로 분할 후 케이스 단위 병합

## 15. 품질 지표 및 테스트셋

필수 정량 지표:
- 4요소 필드 정확도(Exact/Soft match)
- result 상태 분류 정확도(`present/pending/insufficient`)
- evidence span 정합률
- 도메인별(교통/환경/시설/하천) 성능 분해

테스트셋 구성:
- TYPE_A/TYPE_B/TYPE_C 균형 샘플
- 장문/단문/오타/방언/구어체 포함
- answer 없는 사용자 입력 샘플을 최소 30% 이상 확보

## 16. 구현 우선순위 (권장)

1. result 상태값 도입(`pending/insufficient/present`)
2. 세그먼트 분리 강화(Q/A, 민원내용/처리결과 블록)
3. span 생성 로직을 문장 토큰 인덱스 기반으로 변경
4. 요소별 스코어러 도입 및 임계치 튜닝
5. 회귀 테스트(고정 샘플 200건) 자동화

## 17. 운영 가이드

- 규칙은 코드 상수로 하드코딩하지 말고 YAML/JSON 설정으로 외부화
- 규칙 버전(`rule_version`)을 결과에 기록하여 실험 추적성 확보
- 주간 단위 오탐 사례를 수집해 우선순위 큐로 반영

예시 메타:

```json
{
  "rule_version": "v2.0.0",
  "rule_profile": "korean_civil_complaint_default"
}
```

---

이 설계안은 "결과를 억지 생성하지 않는다"는 원칙을 중심으로, 민원 DB(답변 존재)와 사용자 입력(답변 미존재)을 동일 스키마에서 안전하게 처리하도록 구성한다.
