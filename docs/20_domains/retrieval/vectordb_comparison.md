# 벡터DB 후보 비교 메모 (Week 1, BE2)

문서 버전: v1.0  
작성일: 2026-03-17  
담당: BE2 (민건)

## 1. 목적

Week 2 PoC에서 바로 사용할 벡터DB를 1차 고정한다.

비교 대상:
- ChromaDB
- FAISS

## 2. 비교 결과 요약

| 항목 | ChromaDB | FAISS | 1차 판단 |
| --- | --- | --- | --- |
| 설치/초기 적용 | 쉬움 (`pip install chromadb`) | 쉬움 (`faiss-cpu` 또는 `faiss-gpu`) | 동률 |
| 메타데이터 필터 | 네이티브 지원 | 별도 구조 구현 필요 | ChromaDB 우세 |
| API/앱 통합 | Python 앱 통합 단순 | 인덱스/메타 저장 계층 분리 필요 | ChromaDB 우세 |
| 검색 속도 잠재력 | 충분(중소규모) | 매우 높음(대규모) | FAISS 우세 |
| MVP 개발 속도 | 빠름 | 중간(보조 코드 필요) | ChromaDB 우세 |

## 3. 1차 의사결정

- 기본 벡터DB: ChromaDB
- 재검토 조건: Week 2 PoC에서 지연시간 또는 메모리 문제가 반복될 경우

선택 이유:
1. 메타데이터 필터를 바로 적용할 수 있어 FE/QA 요구를 빠르게 충족한다.
2. 현재 팀 규모와 일정에서 구현 리스크가 가장 낮다.
3. 문서와 설정(`configs/models.yaml`) 기준선과 일치한다.

## 4. FAISS를 남겨두는 이유

FAISS는 성능 잠재력이 높아 중기 최적화 옵션으로 유지한다.

적용 고려 시점:
- 인덱스가 커져 검색 지연이 KPI를 위협할 때
- 메타데이터 필터를 별도 저장소로 분리할 여력이 생길 때

## 5. 리스크 및 대응

### 리스크 1: ChromaDB 대량 인덱스 성능 저하
- 징후: 검색 지연이 누적 증가
- 원인: 인덱스 크기 증가/디스크 I/O 병목
- 예방책: 샘플셋으로 성능 로그 주기 측정
- 대응책: 컬렉션 분리, top_k/필터 튜닝
- 최악의 경우 폴백안: FAISS + 메타데이터 보조 인덱스 구성

### 리스크 2: 필터 쿼리 일관성 저하
- 징후: region/category 필터 결과 누락
- 원인: 입력 메타데이터 표준 미고정
- 예방책: BE1-BE2 메타데이터 계약 문서 선고정
- 대응책: 인덱싱 어댑터에서 필드 정규화
- 최악의 경우 폴백안: 필터 없는 유사 검색으로 임시 운영
