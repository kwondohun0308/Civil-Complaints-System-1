# 임베딩 후보 비교 계획 (Week 1, BE2)

문서 버전: v1.0  
작성일: 2026-03-17  
담당: BE2 (민건)

## 1. 목적

Week 2 PoC 이전에 임베딩 모델 1차 선택을 고정하고, 선택 근거를 팀에 공유한다.

- 1차 기준선: BGE-m3
- 대안 후보: KoSimCSE

## 2. 비교 대상

| 항목 | BGE-m3 | KoSimCSE |
| --- | --- | --- |
| 모델명 | `BAAI/bge-m3` | `BM-K/KoSimCSE-roberta` 계열 |
| 언어 범위 | 다국어(한국어 포함) | 한국어 특화 |
| 임베딩 차원 | 1024 | 768 또는 384 (모델별 상이) |
| 장점 | 검색 정확도 기대치 높음, 커뮤니티 사례 풍부 | 상대적으로 가벼움, 한국어 도메인 강점 |
| 리스크 | 임베딩 차원/메모리 비용 큼 | 다국어 일반화 및 장문 검색 안정성 확인 필요 |

## 3. 평가 관점

KPI와 직접 연결되는 항목만 평가한다.

| 관점 | 지표 | 합격선 | 비고 |
| --- | --- | --- | --- |
| 검색 품질 | Recall@5 | 0.85 이상 | 프로젝트 KPI 기준 |
| 지연시간 | 평균 검색 지연(ms) | 질의 E2E 8초 이내 기여 | 임베딩 구간 분리 측정 |
| 안정성 | 연속 2시간 오류율 | 강제 재시작 0회 | OOM/timeout 포함 |
| 구현 용이성 | 적용 난이도 | Week 2 내 적용 가능 | 팀 일정 기준 |

## 4. 실험 계획 (Week 2 착수용)

1. 동일 샘플셋(고정 질의셋)으로 두 모델을 각각 인덱싱한다.
2. `top_k=5` 기준 Recall@5를 측정한다.
3. 인덱싱 시간/검색 시간/메모리 사용량을 함께 기록한다.
4. 결과 표로 비교하고 1차 모델을 고정한다.

측정 스크립트:
- `scripts/evaluate_retrieval.py`
- `scripts/build_index.py`

## 5. 1차 의사결정

현재 문서/설정 기준선과 MVP 범위를 고려해 1차 선택은 아래와 같다.

- 기본 모델: BGE-m3
- 폴백 모델: KoSimCSE

선택 이유:
- 이미 `configs/models.yaml`에서 BGE-m3를 기준선으로 사용 중이다.
- 팀 문서(MVP 범위/역할 문서)와 정합성이 높다.
- Week 2에서 PoC 수치가 기준 미달일 때만 KoSimCSE로 전환 검토한다.

## 6. 리스크 및 대응

### 리스크 1: GPU 메모리 부족
- 징후: 인덱싱 배치 처리 중 OOM 발생
- 원인: 1024차원 임베딩 + 큰 배치
- 예방책: 배치 크기 32 -> 16 -> 8 단계 축소
- 대응책: CPU fallback 또는 야간 배치 분할 실행
- 최악의 경우 폴백안: KoSimCSE로 다운스케일

### 리스크 2: Recall@5 미달
- 징후: 평가셋에서 Recall@5 < 0.85
- 원인: 청크 전략 부적합 또는 도메인 적합성 부족
- 예방책: 청크 텍스트 구성 규칙 고정 후 비교
- 대응책: chunk_type 분리 인덱싱(요약/요청) 실험
- 최악의 경우 폴백안: KoSimCSE 전환 + 하이브리드 검색 검토
